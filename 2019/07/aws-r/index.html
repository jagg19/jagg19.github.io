Running R on an AWS Enstance

July 9, 2019 in AWS, R, MySQL, Programming

Amazon Web Services with R

I went over a bit about AWS in my previous post TK and how benefitial it is. Amazon EC2 is just another great service offered by Amazon. EC2 is basiacally a virtual machine that offers an variety of operating systems and computational power.

EC2 allows users to build apps, automate scaling according to changing needs and peak periods, deploy computational intensive models, streamline development processes, and create virtual servers to manage storage, lessening the need to invest in infrastructure.

Two major advantages of using EC2 is that you aren’t locked into one operating system or one configuration setup. You can go from a standard operating system runninng 1GB ram, to an GPU equipped OS with 100+ GB ram that can handle a wide variety of tasks. You can easily change your instance type in minutes: just stop your instance, change the instance type, and start the instance again to run either more/less power than before. This goes hand in hand with R and RStudio due to it in-memory architecture.


Benefits to R

Memory, memory, and more memory. If you ever cross a dataset thats too large or makes your computer too slow, you can easily pay <$0.80 hour to use an $10,000 GPU that will cut right through whatever dataset your computer was struggling with. Rstudio is in-memory, we can configure the EC to have higher GB for certain uses. There are two ways that we can create an EC2 and install R & Rstudio. The first way is what I will call the easy way, but it is only easy because of the hard work that Louis Aslett put into creating this Rstudio AMI. Louis was kind enough to do all of the dirty work needed (plus more) to create an insntance that will allow us to run Rstudio on AWS. His AMI allows you to get up and running with RStudio and a seamless Dropbox integration in under 5 minutes, so I really appreciate his time for maintaining this AMI!

In order to truely appreciate what he has done, you must go through the next option. The second way is starting from scratch once you create your EC instance. For those who want to get their hands dirty and start from scratch before getting to appreciete the easy way, then jump here. You will have alot more flexibility (and possibly learn more along the way) which is great if you plan on using other applications other than Rstudio. If your looking to create an instance and have Rstudio running all within 5 minutes, then please continue along!


Short and Easy

You have to go through the “long way” to really appreciate how much time your saving by doing it this way. If you just need to quickly create an AWS instance to run RStudio on, then this is the best bet.

Step 1

Click the link for Louis Aslett’s RStudio AMI and view the most recent AMI specs on the right side of the screen. Click the region nearest to you, and a new instance will automatically be created using this AMI, taking you to step 2 of the process.

Step 2

Assuming you are using the free tier, we will select the “General Purpose t2.micro” instance type. Click “Next: Configure Instance Details”

Step 3

Leave all settings to default unless you know which more advanced configuration settings you’d like to set. Click “Next: Add Storage”. You will also leave these settings default unless you believe you may need more storage but make sure to not go above the free tier limit (8GB should be enough). Click “Next: Add Tags”

To help you manage your instances, images, and other Amazon EC2 resources, you can optionally assign your own metadata to each resource in the form of tags. Each tag consists of a key and an optional value, both of which you define. We do not need to set any tags. Click “Next: Configure Security Group”

Step 4

Set your secuirty group name, in this case I will name it “Rstudio” and I will add “rstudio-ami” to the description. Since we need to allow internet traffic to access our RStudio instance, we need to select HTTP under the “Type” drop-down and set the source to custom and enter “0.0.0.0/0, ::/0”. Click “Review and Launch”.

Step 5

Click Launch and wait for the “Select an existing key pair or create a new key pair” tab to open. We will select “Proceed without a key pair” under the first drop-down and click the checkbox that states “I acknowledge that I will not be able to connect to this instance unless I already know the password built into this AMI”. Finally, click Launch Instances!

Step 6

Once your Instance State changes from “Pending” to “running”, select the instance and copy the “Public DNS” and paste it into your browser.

Once your browser connects to the instance, you’ll see a RStudio login screen. The ami’s username is “rstudio” and the password is your instance’s ID. It is highly reccommended to change your password as soon as you login. See below.

Step 7

Now that we are succesfully logged in, let’s change the password. I highly reccomended reading everything in green that Luis Albert wrote, as it explains how to change your password and link to your dropbox. Luckily, it is very easy. First, load the “RStudioAMI” package by running library("RStudioAMI") and then run passwd() which you’ll be prompted to enter the current password, then your new password. After, you can run linkDropbox() to quickly link your dropbox account by entering the link the function outputs into your browser. See below for the Console output after running the above.

Detailed, Longer Way

Step 1: Choosing an AMI

When launching an EC2 instance, you must choose an Amazon Machine Image (AMI), which contains all information required to start an instance. For example, an AMI defines which operating system is installed on your EC2 instance and which software is included.

You can choose the Amazon Linux AMI, which is provided at no additional cost and has a stable version of R in the repository. This AMI is maintained by AWS and includes packages and configurations that provide native integration with AWS and other software. We will use the Ubuntu Server (18.04) AMI to go through the motion of downloading everything from scratch.


Step 2: Choosing an Instance Type

Choose an EC2 instance type that matches the data size and processing that your analysis requires. By default, R runs only on one core node and, in many cases, requires a lot of memory. For programming and development, the general-purpose T2 instance types are sufficient and cheap, and t2.micro is available through the AWS Free Tier. For our purpose, we will select the “General Purpose T2.Micro” Instance. See the following link for more details on AWS EC2 Instance Types.


Step 3: Configuring Instance Details

This is the bread and butter for setting up similar instances each time you run a new instance. Step 4: Configuring the Security Group

In the EC2 launch wizard, you define a security group, which acts as a virtual firewall that controls the traffic for one or more instances. For your R-based analysis environment, you will select HTTP from the type dropdown, and keep the port at 80 for more user friendly capability (Rstudio Server is usually at port 8787). We will set the Source settings to custom with a value of “0.0.0.0/0, ::/0”. We will add an additional rule to allow use to use SSH to login to our instance directly. Select SSH under type, leave the port at 22, and add “0.0.0.0/0” to the source settings. Last, we will add a rule for custom type TCP and add port 3306 with source settings to Anywhere for our MySQL server. After you finish your security group configuration, choose Review and Launch.


Step 5: Logging into RStudio

After your EC2 instance is running, you can connect using a web browser to RStudio Server and R. For login credentials, use the newly created user and password. The URL looks like the following: “http://ec2-YOUR-IP.compute-1.amazonaws.com:8787”

Setting Swap File

Since we are using the free AWS tier which only gives us 1GB of RAM, plentiful in most cases, but some R packages at the moment require a little more RAM than 1GB (TidyVerse). I struggled with this for a little, watching as the computer slowly grinded, before deciding to just bump up the throttle and change my instance to t2.medium from t2.micro. The packages downloaded as I imagined, but I wanted to figure out how to do this without having to leave the free tier. I was using the “top” command when I noticed the EC2 instance was not using any swap memory. As soon as I added 2GB of swap memory to the free tier 1GB t2.micro, all of the packages downloaded without error. If you plan on using instances above t2.medium, then you do not need to do this. Otherwise, see below on how to download packages that require more than 1GB ram on a EC t2.micro instance.

ScreenSHOT!

sudo /bin/dd if=/dev/zero of=/var/swap.1 bs=1M count=2048 sudo /sbin/mkswap /var/swap.1 sudo /sbin/swapon /var/swap.1 sudo sh -c 'echo "/var/swap.1 swap swap defaults 0 0 " >> /etc/fstab
